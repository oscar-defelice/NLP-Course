00

What is NLP?

The field that sits at the intersection of linguistics, statistics, and computation.

Language is the hardest human product to model

Computers handle numbers natively.
Language is ambiguous by design.

"I saw the man with the telescope."
Did I use the telescope to see him?
Or did he have the telescope?
"The bank was steep."
River bank or financial institution?
Context resolves it. Models must learn context.
Ambiguity is everywhere
Lexical · Syntactic · Semantic · Pragmatic
Every level of language has multiple valid interpretations.
Meaning depends on context
Same word, different meanings.
Same sentence, different intentions.
A model that ignores context will fail.

A (brief) history of NLP

1950s–1980s — Symbolic
Hand-crafted rules, grammars, expert systems.
Brittle. Did not scale to real language.
1990s–2010s — Statistical
N-gram models, SVMs, logistic regression.
Learn from data. Language as probability.
Sessions 2–4 of this course.
2013–now — Neural
Word2Vec (2013) → LSTM (2015) → Transformer (2017) → BERT/GPT (2018–).
Sessions 5–8 of this course.
The shift: from hand-crafted rules → learned representations. The same data-driven principle runs through every technique in this course.

NLP Task Taxonomy

The output type defines the task.

The input is always text. The output defines the task.

Classification — Text → label
Sentiment, topic detection, intent, spam.
One label per document.

Input: "This film was absolutely terrible."
Output: negative
Token classification — Text → label per token
Named entity recognition, POS tagging.
One label per word.

Input: "Apple is based in Cupertino."
Output: [ORG] [O] [O] [O] [LOC]
Generation — Text → text
Translation, summarisation, dialogue, completion.
A new sequence as output.

Input: "Bonjour le monde."
Output: "Hello world."
Extraction / QA — Text + query → span
Reading comprehension, information extraction.
A substring or structured record.

Q: "Where is Apple based?"
A: "Cupertino" (span from text)

02

Text Normalisation

Before any model sees your text, it must be cleaned.

Why normalise?

Without normalisation, vocabulary explodes unnecessarily.

Problem:
"Dog", "dog", "DOG" → 3 tokens, 1 concept.
"colour" vs "color" → 2 tokens, 1 concept.
"<br/>", "<p>" → noise masquerading as signal.
Goal: reduce vocabulary without losing meaningful distinctions.
The trade-off
Aggressive normalisation loses information.

"U.S." → "us" may conflate country and pronoun.
Removing punctuation loses "!!!" vs "."
Rule of thumb:
Normalise what is noise.
Preserve what carries meaning.

A standard normalisation pipeline

1
Lowercase — "The" and "the" are the same in most contexts.
2
Strip HTML — <br/>, <p> are markup, not language.
3
Remove punctuation — replace non-alphanumeric with a space.
4
Collapse whitespace — multiple spaces → one space.
5
Strip — remove leading/trailing whitespace.
import re

def preprocess_text(text: str) -> str:
    text = text.lower()
    text = re.sub(
        r"<[^>]+>", " ", text)
    text = re.sub(
        r"[^a-z0-9 ]", " ", text)
    text = re.sub(r" +", " ", text)
    return text.strip()

# Test:
# "Hello, World! <br/>"
# → "hello world"

Optional normalisation steps

Stemming
Heuristic root: strip suffixes.
running → run
flies → fli ⚠

Fast but lossy.
Rarely used in deep learning.
Lemmatisation
Dictionary-based morphological root.
running → run
flies → fly ✓

Slower but correct.
Useful for classical models (S03).
Stop word removal
Remove high-frequency function words:
the, a, is, in, of, …

Reduces noise for BoW.
Can harm sequence models — context matters.
In this course: we use the minimal 5-step pipeline. The TP ablates individual steps so you measure their impact empirically.
03

Tokenisation

Breaking text into the atomic units a model operates on.

Core vocabulary — terms you need to know

Token
The atomic unit of text a model operates on. Depending on the tokeniser, a token can be a word, a subword, a character, or a byte.
"running" may be 1 token or 2 (run + ning).
Type vs Token
Token — every occurrence of a word in a text.
Type — each distinct word form.

"the cat sat on the mat" has 6 tokens, 5 types ("the" appears twice).
Vocabulary V
The complete set of unique types in the corpus.
|V| = vocabulary size — the number of distinct tokens the model knows.
Corpus
The full collection of text used to train or evaluate a model.
Our corpus this session: AG News / IMDb / Emotion.
Document
A single unit of text in the corpus — one news article, one review, one sentence.
The granularity depends on the task.
Out-of-Vocabulary (OOV)
A token seen at inference time but never during training.
Subword tokenisers reduce OOV to near-zero by decomposing unknown words.
Whitespace tokenisers have no OOV strategy.

Three tokenisers, one sentence

"The U.S. economy grew by 2.3% in Q3."
Whitespace split
TheU.S. economygrew by2.3% inQ3.

8 tokens. Punctuation attached to words.

NLTK word_tokenize
TheU.S. economygrew by2.3 %in Q3.

10 tokens. Punctuation split by rules.

GPT-4 BPE
TheĠU .S. Ġeconomy ĠgrewĠby Ġ2.3 %Ġin ĠQ3.

13 tokens. Subword units, data-driven.

Word frequency — the first signal in any text

Before building any model, count things. Frequency is the most basic statistic you can compute on text — and it reveals structure immediately.

What frequency tells you
  • Which words dominate the corpus
  • Which words are rare or unique
  • Whether a preprocessing step worked
  • Whether classes use different vocabularies
Zipf's Law
In any natural language corpus, the frequency of a word is roughly inversely proportional to its rank.

The 1st most frequent word appears ~2× more than the 2nd, ~3× more than the 3rd, and so on.
Consequence: a tiny set of tokens covers most of the text.
Typical top-10 in a news corpus
rank token freq useful?
1the84,230✗ stop word
2of51,400✗ stop word
3in43,100✗ stop word
.........
41said8,900~ depends
89percent4,200✓ Business
142goal2,100✓ Sports

The top tokens are function words — grammatically necessary but class-neutral. The informative tokens sit further down the frequency list.

Frequency as a feature — why it works for classification

Once you have tokens, the simplest question you can ask is: how often does each token appear?

Document A — Sports
goalgoalmatch teamplayerscore theina
Document B — Politics
electionvotesenate presidentpolicy theina

Each document becomes a count vector over the vocabulary:

A = [goal:2, match:1, team:1, election:0, vote:0, ...]
B = [goal:0, match:0, team:0, election:1, vote:1, ...]
A classifier learns that high goal count → Sports.
High election count → Politics.
The text has become a point in |V|-dimensional space.
Term Frequency (TF)
How often token t appears in document d.
TF(t,d) = count(t in d)
Document Frequency (DF)
How many documents in the corpus contain token t.
High DF = common across all classes = less informative.
TF-IDF
Downweights tokens frequent everywhere, upweights tokens distinctive to a document.
Formula and worked example — next two slides.

TF-IDF — the formula

Raw counts have a problem: "the" appears 50,000 times and "goal" appears 200 times, but "goal" is far more informative. TF-IDF corrects for this.

Term Frequency — TF
How often token t appears in document d. Normalised by document length to make short and long documents comparable.
TF(t, d) = count(t in d) / |d|
Inverse Document Frequency — IDF
Downweights tokens that appear in many documents — they carry little class-specific information.
IDF(t) = log ( N / (1 + DF(t)) )
N = total documents. +1 avoids division by zero.
TF-IDF score
The product of the two.
TF-IDF(t, d) = TF(t, d) · IDF(t)
High score = token appears often in this document but rarely across the corpus.
That is the signature of a discriminative word.
Intuition:
"the" → high TF, but also high DF → IDF ≈ 0 → TF-IDF ≈ 0
"goal" → moderate TF, low DF in non-sports docs → high TF-IDF in sports documents

TF-IDF — worked example

Corpus of 4 documents. We compute TF-IDF for two tokens: goal and the.

doc label words "goal" "the"
D1 Sports 100 4 12
D2 Politics 80 0 10
D3 Politics 120 0 15
D4 Sports 90 3 11
IDF — computed once over the corpus
N = 4
DF("goal") = 2 (D1, D4)
DF("the") = 4 (all docs)

IDF("goal") = log(4 / 3) ≈ 0.29
IDF("the") = log(4 / 5) ≈ −0.22 → clipped to 0
TF-IDF("goal", D1)
TF = 4/100 = 0.04
IDF = 0.29
TF-IDF = 0.04 × 0.29 = 0.012
Non-zero — "goal" is distinctive for D1.
TF-IDF("the", D1)
TF = 12/100 = 0.12
IDF ≈ 0 (appears in every doc)
TF-IDF = 0.12 × 0 = 0.000
Zero — "the" is useless as a feature.

The full document vector for D1 is a |V|-dimensional array. Most entries are 0. A classifier trained on these vectors learns to weight the non-zero, high-IDF tokens.

The vocabulary size trade-off

Large vocabulary
+ Each token is more meaningful
+ Shorter sequences
− Many rare / unseen words (OOV)
− Huge embedding matrix to learn
Small vocabulary (character-level)
+ Zero OOV rate
+ Tiny embedding matrix
− Very long sequences
− Single characters carry little meaning
The sweet spot: subword
Frequent words → one token.
Rare words → split into subword units.

unhappiness
→ un + happi + ness
GPT-4: ~100,000 tokens
BERT: ~30,000 tokens
Chars: 256 bytes

Byte-Pair Encoding (BPE)

A data-driven algorithm. The vocabulary reflects corpus frequency, not grammar rules.

1
Initialise — every character is a token. Vocabulary = all unique characters.
2
Count — find the most frequent adjacent token pair in the training corpus.
3
Merge — combine that pair into a new token. Add to vocabulary.
4
Repeat — until vocabulary reaches target size (e.g. 50,000 merges for GPT-2).
l o w → lo w → low → low er 
import tiktoken

enc = tiktoken.encoding_for_model(
    "gpt-4"
)
tokens = enc.encode(
    "unhappiness"
)
print(tokens)
# [1906, 81, 1918, 1481]

decoded = [
    enc.decode([t])
    for t in tokens
]
print(decoded)
# ['un','h','app','iness']
04

Dataset Hygiene

How to not fool yourself — or your model.

Train / Validation / Test

Train set
The model learns from this.
Typically 70–80% of data.

Used in every gradient step.
Validation set
You tune hyperparameters on this.
Typically 10–15%.

Used to select config. Never train on it.
Test set
You report results from this.
Typically 10–20%.

Touch exactly once. At the very end.
Golden rule: never use test set performance to make any decision. The moment you do, it becomes a second validation set and your numbers are optimistic.

Data leakage

Leakage = information from val/test contaminates training.

Classic mistake:
Fit TF-IDF on the full dataset, then split.
The vocabulary has seen test documents.
Your model knows things it should not.
Correct order:
Split → fit preprocessing on train only → transform val/test with those parameters.
1. Split
↓
2. Fit on train only
↓
3a. Transform train
3b. Transform val
3c. Transform test

scikit-learn's Pipeline enforces this automatically.

Stratified splitting

Random splitting can misrepresent rare classes, especially in small val/test sets.

Without stratification:
A rare class (5% of data) might appear
0 times in a small validation set.
Your val metrics are meaningless for that class.
With stratification:
Every split has the same class proportions
as the full dataset. Evaluation is representative. 
from sklearn.model_selection \
    import train_test_split

train_df, val_df = train_test_split(
    df,

Evaluation & Baselines

Accuracy is not enough. Here is why.

The confusion matrix

For binary classification, four possible outcomes:

Predicted +Predicted −
Actual + TP
True Positive		 FN
False Negative
Actual − FP
False Positive		 TN
True Negative
Accuracy = (TP + TN) / (TP+TN+FP+FN)
Precision = TP / (TP + FP)
Recall   = TP / (TP + FN)
Precision: of all my positive predictions, how many were correct?
Recall: of all actual positives, how many did I find?

Why accuracy fails on imbalanced data

Scenario: spam detection. 95% of emails are legitimate, 5% are spam.
A classifier that always predicts "not spam" gets 95% accuracy.
It is completely useless — it catches zero spam.
F1 Score — the harmonic mean of P and R
F1 = 2 · (P · R) / (P + R)
Both P and R must be high for F1 to be high.
Penalises models that sacrifice one for the other.
Macro vs Weighted F1
Macro F1: average F1 across classes, equally weighted.
Treats rare and frequent classes the same.

Weighted F1: weighted by class support.
Reflects production distribution.

Report macro F1 when all classes matter equally. Report weighted F1 for overall production performance. Always report both.

The majority-class baseline

Find the most frequent class in the training set.
Predict that class for every input.
Ignore the text entirely.
What it tells you
Balanced K classes → baseline accuracy = 1/K
Imbalanced (90% majority) → 90% accuracy
In the second case, accuracy is misleading.
Macro F1 of baseline is always very low.
Only one class has non-zero recall.
This is the honest number to beat. 
from collections import Counter
from sklearn.metrics import \
    classification_report

def majority_baseline(train_df, eval_df):
    majority = (
        Counter(train_df["label"])
        .most_common(1)[0][0]
    )
    y_true = eval_df["label"].tolist()
    y_pred = [majority] * len(y_true)
    return y_true, y_pred

y_true, y_pred = majority_baseline(
    train_df, test_df
)
print(classification_report(
    y_true, y_pred,
    zero_division=0
))
