The Problem

A company has thousands of resolved IT tickets. Every day, users open new tickets for problems that have already been solved.

Without a chatbot

  • User faces an error
  • Opens a new ticket
  • Waits hours for a reply
  • Agent finds the same answer from last month
  • Cost: agent time + user frustration

With a chatbot

  • User describes the error
  • System retrieves similar past tickets
  • LLM synthesises a clear answer
  • User resolves the issue in seconds
  • Ticket opened only if needed

This is a real production pattern used by Notion, Intercom, Slack, and most large IT teams.

What is RAG?

Retrieval-Augmented Generation · Lewis et al., 2020

Core idea: instead of asking an LLM to memorise all knowledge at training time, give it relevant documents at inference time via a retrieval step.

Why not just use the LLM directly?
LLMs hallucinate, have a knowledge cutoff, and cannot access your private KB.
Why not just do keyword search?
Search returns documents; RAG synthesises an answer grounded in those documents.

Two Phases: Offline vs Online

Offline (indexing)

Done once, before the chatbot runs.
1. Load resolved tickets
↓
2. Clean & normalise text
↓
3. Build index (TF-IDF or embeddings)
↓
4. Store index on disk

Online (query)

Runs for every user message.
1. Receive user query
↓
2. Embed / vectorise query
↓
3. Retrieve top-k tickets
↓
4. Build prompt + call LLM
↓
5. Return answer to user

Retriever v1: TF-IDF + Cosine Similarity

How it works

  • Represent each ticket as a sparse vector of TF-IDF weights
  • At query time, vectorise the query with the same fitted vectoriser
  • Rank tickets by cosine similarity to the query
  • Return top-k
Limitation: vocabulary mismatch.
"WiFi won't work" ≠ "network connectivity issue" -- zero overlap in TF-IDF space.

Worked example

Query
"printer not responding after update"

Top-3 retrieved
① 0.81 "printer stops working after driver update..."
② 0.63 "device not responding, tried updating..."
③ 0.44 "software update caused printer queue to freeze..."

Retriever v2: Sentence Embeddings

Model: all-MiniLM-L6-v2 (sentence-transformers) · 384-dim dense vectors · 80 MB · CPU-friendly

What changes?

  • Each text → a dense 384-dim vector encoding meaning
  • Semantically similar sentences are close in vector space
  • Retrieval still uses cosine similarity
  • Index is now a matrix (n_tickets, 384)

Same example

Query
"my printer stopped after the update"

Top match (TF-IDF missed it)
① 0.89 "after installing the latest OS patch, the HP LaserJet became unresponsive"
(zero lexical overlap with query!)
Key insight: embeddings capture semantic similarity; TF-IDF captures lexical overlap. In production, both are often combined (hybrid search).

Prompt Engineering for RAG

The retrieved tickets go into the prompt as grounding context.

SYSTEM: You are an IT support assistant. Answer the user's question using ONLY the context below. If the context does not contain enough information, say so and suggest opening a ticket.

CONTEXT (resolved tickets):
--- Ticket 1 (Network) ---
Issue: My laptop won't connect to WiFi after a Windows update.
Solution: Rollback the network driver via Device Manager > Network Adapters > right-click > Update Driver > Browse > Let me pick.

--- Ticket 2 (Network) ---
Issue: WiFi keeps dropping every few minutes.
Solution: Disable power saving for the adapter in Advanced Settings.

USER: My WiFi stopped working after I updated Windows yesterday.
Design choices: tell the model to stay grounded · instruct it to admit uncertainty · always include the source ticket category

Gradio: Deploy in 10 Lines

Gradio wraps any Python function into a web UI. No HTML, no JavaScript. 
import gradio as gr

def chat_fn(user_message, history):
    result = bot.answer(user_message)
    sources = "\n".join(
        f"[{s['category']}] {s['issue'][:80]}…"
        for s in result["sources"]
    )
    return result["response"] + f"\n\n---\n📎 Sources:\n{sources}"

demo = gr.ChatInterface(
    fn=chat_fn,
    title="IT Helpdesk Assistant",
    description="Describe your issue and I'll search our resolved tickets.",
)
demo.launch()
Running in Colab: demo.launch(share=True) gives a public URL valid for 72h -- no account needed.

Evaluation

RAG systems are harder to evaluate than classifiers -- there is no single ground-truth label.

Retrieval quality

For a given query, does the correct ticket appear in top-k?

Metric: Recall@k
Answer quality

Is the LLM answer grounded in the retrieved context?

Metric: faithfulness (manual or LLM-as-judge)
End-to-end

Did the user get a useful answer without opening a ticket?

Metric: deflection rate (production KPI)
In the TP: compare TF-IDF vs embedding retriever on 10 hand-crafted test queries. For each, check whether the most relevant ticket is in top-3.
